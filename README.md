# 研究設備予約システム：リファクタリング演習（Python版・TEAM-B）

対象：3名グループ

大学の共用研究設備を予約し，利用料金を計算する小さなPythonプログラムです．
現行仕様については動作し，付属のテストもすべて通ります．一方，コードには変更を難しくする構造が複数含まれています．

## 演習の目的

この演習では，現在の外部仕様を維持したまま，コードを変更しやすい構造へ段階的に改善します．
完成形を当てることより，次の点を重視します．

- どこに設計上の問題があると判断したか
- その問題が，どのような仕様変更を難しくするか
- どの責務や概念を，どこへ移したか
- 改善のために増やした複雑さが妥当か
- テストを使い，どのように安全に変更したか

Pythonでは，すべてをクラスや継承で表現する必要はありません．関数，`dataclass`，`Enum`，プロトコル，モジュール分割なども含め，目的に合う方法を選んでください．

## 実行環境

- Python 3.11以上
- Bashを利用できる環境（Linux，macOS，WSL，Git Bashなど）

WindowsのコマンドプロンプトまたはPowerShellでは `test.bat` も利用できます．
外部ライブラリは不要です．

## テストの実行

```bash
./test.sh
```

Windowsでは次を実行します．

```text
test.bat
```

直接実行する場合は次の通りです．

```bash
PYTHONPATH=src python3 -m unittest discover -s tests -v
```

[docs/current-specification.md](docs/current-specification.md) を参照してください．

## 課題

1. 現在のコードを読み，設計上の問題をIssueに整理してください．
2. 既存テストを実行し，現在の振る舞いを確認してください．
3. 不足している仕様化テストを追加してください．
4. 現在の外部仕様を維持したまま，段階的にリファクタリングしてください．
5. Pull Requestに，主要な設計判断と変更前後の違いを記述してください．
6. グループ内でレビューし，指摘への対応を残してください．

## 制約

- 現在の仕様を変更しないでください．
- 既存テストを削除したり，期待値を都合よく変更したりしないでください．
- 将来ありそうな機能を推測して先に実装しないでください．
- 一度に全面的に書き換えず，意味の分かる小さなコミットに分けてください．
- クラス数，抽象基底クラス数，デザインパターンの数を増やすこと自体を目的にしないでください．
- 型ヒントの追加だけで，設計上の問題が解消したと判断しないでください．

## 推奨するGitHub上の進め方

- 問題点ごとにIssueを作成する
- Issue単位または小さな改善単位でブランチを作成する
- Pull Requestを作成し，別メンバーがレビューする
- リファクタリングと機能追加を同じコミットに混ぜない
- PR本文には「何を変えたか」だけでなく「なぜ変えたか」を書く

## 発表で説明する内容

- 元のコードで，特に重要だと判断した問題
- 想定した変更と，元の構造での影響範囲
- 実施したリファクタリング
- テストをどのように利用したか
- 改善された点と，複雑になった点
- 採用しなかった案と，その理由
- グループ内レビューで変わった点
